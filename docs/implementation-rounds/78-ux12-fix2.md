## 1. STALE-SERVING DIAGNOSIS

### Serving chain — 3 live paths at diagnosis time

| Port | PID | Started | Command | Reload? | Serves |
|------|-----|---------|---------|---------|--------|
| 3015 | 9516 | earlier | `python3 -m http.server 3015 --bind 0.0.0.0` | N/A | `docs/mocks/` (design mockups, not app) |
| 3000 | 11404 | 17:58 | `uvicorn app.main:app --host 0.0.0.0 --port 3000` | NO | API + SPA from `dist/` |
| 8000 | 18234 | 19:22 | `uvicorn app.main:app --host 0.0.0.0 --port 8000` | NO | API + SPA from `dist/` |

### Root cause

Both uvicorn instances were started **before** any UX12 changes (19:47+). Without `--reload`, Python modules are cached in memory at startup:

- `app/main.py` changes adding `"date"` to snapshots → **NOT reflected** in either process
- `app/main.py` changes adding `"unresolved"` to claim_summary → **NOT reflected** in either process

The SPA fallback (`app/main.py:627`) returns `FileResponse(dist/index.html)` — this reads from disk, so frontend JS was fresh after `npm run build`. But the stale API responses meant the fresh frontend received responses missing `date` and `unresolved` fields.

Also: two identical services on two ports created confusion about which URL was "the app."

### Fix applied

```
kill 11404 18234 9516  # killed all three stale processes
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload  # single instance with auto-reload
```

**ONE URL:** `http://localhost:8000` — serves both API endpoints and SPA frontend. `--reload` ensures subsequent code changes are reflected instantly without restart.

## 3. VfTrend LEGIBILITY — Chart.js Tick Colors

### Root cause

Chart.js renders to HTML Canvas via Canvas2D API. The component code used CSS custom property strings directly:

```ts
ticks: { color: "var(--nn-text-dim)" }     // → canvas fillStyle = "var(--nn-text-dim)"
grid: { color: "var(--nn-border)" }         // → canvas strokeStyle = "var(--nn-border)"
dataset: { borderColor: "var(--nn-teal)" }  // → same problem
```

Canvas2D's `fillStyle` and `strokeStyle` accept CSS color values, but **do NOT resolve CSS custom properties** (`var(--...)`). The browser treats `"var(--nn-text-dim)"` as an unrecognized color → falls back to `#000000` (black).

On dark mode background (`#0c0f0b`), black text on near-black = invisible. The Radar chart had the same defect — tick labels, axis labels, grid lines, legend labels all rendering as black.

### Fix

Added two helpers in each Chart.js component:

```ts
/** Resolve CSS custom property to hex for Canvas2D rendering. */
function cssVar(name: string): string {
  if (typeof document === "undefined") return "#888";
  return getComputedStyle(document.documentElement)
    .getPropertyValue(name).trim() || "#888";
}

/** Convert hex to rgba with alpha. */
function hexToRgba(hex: string, alpha: number): string {
  const h = hex.replace("#", "");
  const r = parseInt(h.slice(0,2), 16);
  const g = parseInt(h.slice(2,4), 16);
  const b = parseInt(h.slice(4,6), 16);
  return `rgba(${r},${g},${b},${alpha})`;
}
```

Resolved once at render time, used everywhere:

```ts
const teal = cssVar("--nn-teal");        // dark: #5ebd8e, light: #276b52
const text = cssVar("--nn-text");         // dark: #d2e4c5, light: #1c2018
const textDim = cssVar("--nn-text-dim");  // dark: #738567, light: #717a68
const border = cssVar("--nn-border");     // dark: #2c3625, light: #d0d5c7
```

This works in both light and dark mode — the resolved hex matches the active theme.
